## Key design decisions

**Web never touches the database or connectors directly.**
All data access goes through `api` over REST or Server-Sent Events. The `NEXT_PUBLIC_API_BASE_URL` environment variable controls which `api` instance the web layer talks to (defaults to `http://localhost:8000`).

**Auth lives in `web`, session validation in `api`.**
Auth.js (NextAuth) handles the GitHub OAuth flow and session cookie in the Next.js layer. Every API call from the browser includes the JWT; `api` validates it with the `jwt_secret` from its config.

**Graph visualization uses React Flow.**
The blast-radius graph is rendered as an interactive force-directed graph in the browser, built on React Flow. Graph data (nodes and edges) is fetched from the `api` and rendered client-side.

### SSE live updates (`use-sse.ts`)

`EventSource` doesn't support custom headers, so the hook uses `fetch` + `ReadableStream` to connect to `GET /workspaces/{id}/events` with a `Bearer` token. It:
- Auto-reconnects with exponential back-off (capped at 30 s)
- Parses `data:` lines as JSON, skipping keep-alive comments
- Fires `onEvent` callbacks, which trigger React Query `invalidateQueries`

### Entry points

| User action | Route |
|---|---|
| Click "Rotate Secret" on blast-radius graph | `POST /secrets/{id}/rotate` в†’ redirect to `/rotations` |
| Approve plan | `POST /rotations/{id}/approve` в†’ engine enqueued |
| Confirm step gate | `POST /rotations/{id}/steps/{step_id}/confirm` в†’ engine re-enqueued |
| Simulate failure (demo) | `POST /demo/session/{id}/simulate-failure` в†’ next verify step fails в†’ rollback |

### Audit log

`/audit` page queries `GET /workspaces/{id}/audit` (newest-first) and shows:
- Action name (color-coded by action category)
- Actor (JWT sub / `system`)
- Relative timestamp + absolute on hover
- Truncated SHA-256 chain hash for tamper-evidence visibility

## Running the web UI

```bash
# Install dependencies (first time)
make web-install    # runs: cd web && pnpm install

# Start dev server
make web-dev        # runs: cd web && pnpm dev
# в†’ http://localhost:3000
```

The web Dockerfile is only used for production/hosted-demo builds, not development. In dev, pnpm runs the Next.js dev server directly with hot reload.
