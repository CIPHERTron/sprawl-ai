## Role

The `worker` service is where all **non-trivial computation** happens. The API never runs agents or multi-step rotation logic on a request thread ‚Äî it only enqueues jobs. The worker dequeues and executes them.

Worker responsibilities:
- Repo history scans (call gitleaks / detection library, process results) ‚Äî M8
- Post-processing detections into `Secret` + `Finding` rows ‚Äî M8
- LangGraph agent runtime: **Investigator** agent (blast-radius mapping + severity) ‚Äî **M5 ‚úì**
- LangGraph agent runtime: **Rotation Planner** agent ‚Äî M6
- Deterministic rotation state machine execution (step-by-step, human-gated) ‚Äî **M5 ‚úì**
- Publishing progress events to Redis pub/sub so `api` can stream them to the browser ‚Äî **M5 ‚úì**

## Entry point ‚Äî `worker/main.py`

arq discovers the worker via `WorkerSettings`:

```python
class WorkerSettings:
    functions = JOB_REGISTRY     # list of arq job functions
    on_startup = startup         # also calls setup_checkpointer() for LangGraph DDL
    on_shutdown = shutdown

    redis_settings = RedisSettings.from_dsn(worker_settings.redis_url)
    job_timeout = 300            # 5-minute default
    max_tries = 3
    keep_result = 3600           # keep job results for 1 hour
```

The Docker entrypoint runs: `arq worker.main.WorkerSettings`

## Job registry ‚Äî `worker/jobs/`

`worker/jobs/__init__.py` exports `JOB_REGISTRY` ‚Äî a list of all arq job functions. Adding a new job means:
1. Create the function in a file under `worker/jobs/`.
2. Import and add it to `JOB_REGISTRY` in `worker/jobs/__init__.py`.
3. From `api`, enqueue via `await arq_pool.enqueue_job("function_name", ...)`.

## LangGraph investigation agent (M5)

The full investigation pipeline lives in `worker/agents/`. It uses a **StateGraph** with five nodes:

```
ingest ‚Üí triage ‚Üí [investigator ‚Üí blast_radius ‚Üí severity] ‚Üí END
                ‚Üò‚Üí END  (if triage_score < 20 and not high-value type)
```

### Node responsibilities

| Node | File | What it does |
|---|---|---|
| `ingest` | `nodes/ingest.py` | Reads `secrets`, `findings`, `connectors` from DB; populates `SecretContext` |
| `triage` | `nodes/triage.py` | Cheap heuristic score from exposure status + type; decides `"investigate"` vs `"skip"` |
| `investigator` | `nodes/investigator.py` | Calls GitHub (file snippet), Vault (verify store), IAM (GetAccessKeyLastUsed) concurrently |
| `blast_radius` | `nodes/blast_radius.py` | `IamConnector.enumerate_scope` + `BlastRadiusBuilder`; writes in-memory graph nodes/edges |
| `severity` | `nodes/severity.py` | Deterministic scorer (0‚Äì100) + optional LLM explanation via Ollama |

### Supporting packages

| Package | Purpose |
|---|---|
| `worker/connectors/` | Vault (hvac AppRole), AWS IAM (boto3 + AssumeRole), GitHub (httpx) |
| `worker/severity/` | Deterministic `SeverityEngine` ‚Äî scope √ó env √ó exposure formula |
| `worker/blastradius/` | `BlastRadiusBuilder` ‚Äî assembles `GraphNodeData` + `GraphEdgeData` |
| `worker/llm/` | `litellm.acompletion` wrapper + Langfuse span; gracefully returns `None` if Ollama unavailable |
| `worker/agents/guardrails.py` | Per-node timeout + tenacity retry + error capture decorator |

### Checkpointing

LangGraph uses `AsyncPostgresSaver` (psycopg3) for checkpoint persistence:
- `thread_id = investigation_id`
- DDL (`setup()`) runs once at `worker.startup`
- Checkpoint is created on every node transition, enabling resume on crash

## Rotation state machine (M5 ‚úì)

The rotation state machine is a **deterministic**, audited, idempotent Postgres-backed engine ‚Äî not an LLM. It lives in `worker/rotation/`.

```
pending_approval ‚Üí provisioning ‚Üí distributing ‚Üí [awaiting_confirmation]
  ‚Üí verifying ‚Üí [awaiting_confirmation] ‚Üí revoking ‚Üí completed
                                       ‚Üò‚Üí rolling_back ‚Üí rolled_back
                                                       ‚Üò‚Üí rollback_failed
```

### Rotation packages

| Package / File | Role |
|---|---|
| `worker/rotation/engine.py` | `advance_rotation()` ‚Äî core loop; reads DB, executes next step, persists result, re-enqueues |
| `worker/rotation/steps.py` | `execute_step()` + `compensate_step()` ‚Äî one function per StepKind |
| `worker/rotation/gate.py` | `verify_gate` + `coverage_gate` ‚Äî enforce verify-before-revoke (I1) and coverage (I2) |
| `worker/rotation/lock.py` | `RotationLock` ‚Äî Redis `lock:rotation:{secret_id}` SET NX with 5-min TTL |
| `worker/rotation/sandbox.py` | `SandboxStoreConnector` + `SandboxCloudConnector` ‚Äî in-memory connectors for demo/tests |
| `worker/jobs/run_rotation_step.py` | arq job ‚Äî acquires lock, calls `advance_rotation`, re-enqueues if `should_continue=True` |

### Safety invariants enforced

| ID | Invariant | Where enforced |
|---|---|---|
| I1 | Verify-before-revoke | `verify_gate()` in `gate.py` ‚Äî blocks revoke until all verify steps are `done` |
| I2 | Coverage gate | `coverage_gate()` in `gate.py` ‚Äî blocks revoke if `unknown_consumers > 0` and user hasn't confirmed |
| I3 | Plan TTL | `engine.py` ‚Äî expires `pending_approval` rotations whose `plan_expires_at` has passed |
| I4 | Auto-rollback on failure | `engine._rollback()` ‚Äî runs `compensate_step` on all `done` steps in reverse order |
| I5 | Idempotency | `_find_next_step()` ‚Äî skips all steps not in `pending` state |
| I6 | Confirmation gate | Engine pauses (`awaiting_confirmation`) when `requires_confirmation=True`; resumes after API `confirm_step` |

### Connector isolation (C1)

`CredentialMaterial` returned by `create_credential` is held in-memory only:
- written to the secret store (Vault/sandbox) and immediately discarded
- never persisted to PostgreSQL
- never logged
- never sent to an LLM

### Sandbox mode

`build_sandbox_connectors(fail_at=None)` creates pure in-memory connectors with no external calls. Pass `fail_at="revoke"` to inject a controlled failure for the demo simulate-failure flow.

See [Verify before revoke](/architecture/verify-before-revoke) for the full safety model.

## Vault credential access

Only `worker` (and `api`) read from Vault. The worker authenticates via AppRole using `VAULT_ROLE_ID` and `VAULT_SECRET_ID` from the environment. Connector credentials are fetched from Vault at job execution time and held in memory ‚Äî never logged or persisted to PostgreSQL.
