## Role

The `worker` service is where all **non-trivial computation** happens. The API never runs agents or multi-step rotation logic on a request thread — it only enqueues jobs. The worker dequeues and executes them.

Worker responsibilities:
- Repo history scans (call gitleaks / detection library, process results) — M8
- Post-processing detections into `Secret` + `Finding` rows — M8
- LangGraph agent runtime: **Investigator** agent (blast-radius mapping + severity) — **M5 ✓**
- LangGraph agent runtime: **Rotation Planner** agent — M6
- Deterministic rotation state machine execution (step-by-step, human-gated) — M6
- Publishing progress events to Redis pub/sub so `api` can stream them to the browser — **M5 ✓**

## Entry point — `worker/main.py`

arq discovers the worker via `WorkerSettings`:

```python
class WorkerSettings:
    functions = JOB_REGISTRY     # list of arq job functions
    on_startup = startup         # also calls setup_checkpointer() for LangGraph DDL
    on_shutdown = shutdown

    redis_settings = RedisSettings.from_dsn(worker_settings.redis_url)
    job_timeout = 300            # 5-minute default
    max_tries = 3
    keep_result = 3600           # keep job results for 1 hour
```

The Docker entrypoint runs: `arq worker.main.WorkerSettings`

## Job registry — `worker/jobs/`

`worker/jobs/__init__.py` exports `JOB_REGISTRY` — a list of all arq job functions. Adding a new job means:
1. Create the function in a file under `worker/jobs/`.
2. Import and add it to `JOB_REGISTRY` in `worker/jobs/__init__.py`.
3. From `api`, enqueue via `await arq_pool.enqueue_job("function_name", ...)`.

## LangGraph investigation agent (M5)

The full investigation pipeline lives in `worker/agents/`. It uses a **StateGraph** with five nodes:

```
ingest → triage → [investigator → blast_radius → severity] → END
                ↘→ END  (if triage_score < 20 and not high-value type)
```

### Node responsibilities

| Node | File | What it does |
|---|---|---|
| `ingest` | `nodes/ingest.py` | Reads `secrets`, `findings`, `connectors` from DB; populates `SecretContext` |
| `triage` | `nodes/triage.py` | Cheap heuristic score from exposure status + type; decides `"investigate"` vs `"skip"` |
| `investigator` | `nodes/investigator.py` | Calls GitHub (file snippet), Vault (verify store), IAM (GetAccessKeyLastUsed) concurrently |
| `blast_radius` | `nodes/blast_radius.py` | `IamConnector.enumerate_scope` + `BlastRadiusBuilder`; writes in-memory graph nodes/edges |
| `severity` | `nodes/severity.py` | Deterministic scorer (0–100) + optional LLM explanation via Ollama |

### Supporting packages

| Package | Purpose |
|---|---|
| `worker/connectors/` | Vault (hvac AppRole), AWS IAM (boto3 + AssumeRole), GitHub (httpx) |
| `worker/severity/` | Deterministic `SeverityEngine` — scope × env × exposure formula |
| `worker/blastradius/` | `BlastRadiusBuilder` — assembles `GraphNodeData` + `GraphEdgeData` |
| `worker/llm/` | `litellm.acompletion` wrapper + Langfuse span; gracefully returns `None` if Ollama unavailable |
| `worker/agents/guardrails.py` | Per-node timeout + tenacity retry + error capture decorator |

### Checkpointing

LangGraph uses `AsyncPostgresSaver` (psycopg3) for checkpoint persistence:
- `thread_id = investigation_id`
- DDL (`setup()`) runs once at `worker.startup`
- Checkpoint is created on every node transition, enabling resume on crash

## Rotation state machine (M6+)

The rotation state machine is a **deterministic**, audited, idempotent Postgres-backed engine — not an LLM. It:
- Reads the current `Rotation` and its `RotationStep` rows
- Executes one step at a time using the appropriate connector
- Writes step outcomes (status, executed_at, error) back to PostgreSQL
- Publishes progress events to Redis pub/sub
- Runs compensation (rollback) in reverse order if any step fails

See [Verify before revoke](/architecture/verify-before-revoke) for the full safety model.

## Vault credential access

Only `worker` (and `api`) read from Vault. The worker authenticates via AppRole using `VAULT_ROLE_ID` and `VAULT_SECRET_ID` from the environment. Connector credentials are fetched from Vault at job execution time and held in memory — never logged or persisted to PostgreSQL.
